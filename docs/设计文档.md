# 受试者随访提醒与访视窗口计算系统 - 设计文档

## 一、需求分析

### 1.1 项目背景
某三甲医院药物临床试验机构，开展肿瘤药物III期临床试验，入组120名受试者。每位受试者需完成10个访视节点：筛选期→治疗期（6个周期）→随访期（3次）。

### 1.2 核心痛点
- 手动计算访视日期易出错
- 错过访视窗口导致方案偏离
- CRC人工核对耗时

### 1.3 项目目标
开发轻量级Web工具，自动计算访视窗口，提供每日提醒清单。

---

## 二、业务规则分析

### 2.1 访视节点规则

| 访视节点 | 时间规则 | 窗口期 | 类型 |
|---------|---------|--------|------|
| 筛选期 (Screening) | 入组当天 D0 | ±0天 | 关键节点 |
| 治疗期C1 | D0 + 21天 | ±3天 | 关键节点 |
| 治疗期C2 | C1 + 21天 | ±3天 | 关键节点 |
| 治疗期C3 | C2 + 21天 | ±3天 | 关键节点 |
| 治疗期C4 | C3 + 21天 | ±3天 | 关键节点 |
| 治疗期C5 | C4 + 21天 | ±3天 | 关键节点 |
| 治疗期C6 | C5 + 21天 | ±3天 | 关键节点 |
| 随访期F1 | C6 + 28天 | ±7天 | 允许窗口宽松 |
| 随访期F2 | F1 + 28天 | ±7天 | 允许窗口宽松 |
| 随访期F3 | F2 + 28天 | ±7天 | 允许窗口宽松 |

### 2.2 特殊规则
1. **实际日期回填重算**：若某个访视实际发生日期偏离计划日期，后续所有访视需以实际发生日期重新计算
2. **方案偏离标记**：超出窗口期未完成访视，标记为"方案偏离"
3. **提前退出**：受试者若提前退出试验（脱落），停止所有后续提醒

## 三、技术架构

### 3.1 技术选型
- **后端**: Node.js + Express.js
- **数据库**: SQLite (sql.js — 基于 WASM，无需编译原生模块)
- **日期处理**: dayjs
- **Excel处理**: xlsx + multer
- **测试**: Jest + supertest

### 3.2 架构模式
采用经典MVC三层架构：
```
Controller (路由层) → Service (业务逻辑层) → Repository (数据访问层)
```

## 六、模块功能详细设计

### 6.1 工具层 (utils/)

#### dateCalculator.js - 日期计算核心
- `calculatePlannedDate(baseDate, offsetDays)` → 计算计划日期
- `calculateWindowDates(plannedDate, windowDays)` → 计算窗口期 [start, end]
- `isDateInWindow(date, windowStart, windowEnd)` → 判断日期是否在窗口内
- `isOverdue(plannedDate, windowEnd, referenceDate)` → 判断是否超窗
- `daysUntilDeadline(windowEnd, referenceDate)` → 计算距截止日剩余天数
- `recalculateSubsequentVisits(visits, fromIndex, actualDate)` → 以实际日期重算后续访视

#### validators.js - 数据校验
- `validateSubject(data)` → 校验受试者数据
- `validateVisit(data)` → 校验访视数据
- `isValidDate(dateStr)` → 校验日期格式

### 6.2 数据访问层 (repositories/)

#### subjectRepository.js
- `create(data)` → 创建受试者
- `findById(id)` → 按ID查询
- `findAll(filters)` → 查询列表（支持筛选）
- `update(id, data)` → 更新受试者
- `delete(id)` → 删除受试者
- `findBySubjectNo(subjectNo)` → 按编号查询

#### visitRepository.js
- `create(data)` → 创建访视记录
- `bulkCreate(visits)` → 批量创建访视记录
- `findById(id)` → 按ID查询
- `findBySubjectId(subjectId)` → 查询受试者的所有访视
- `update(id, data)` → 更新访视记录
- `findPendingByDateRange(startDate, endDate)` → 按日期范围查询待完成访视
- `findOverdueVisits(date)` → 查询超窗访视

#### deviationRepository.js
- `create(data)` → 创建偏离记录
- `findBySubjectId(subjectId)` → 按受试者查询
- `findAll(filters)` → 查询所有偏离记录
- `getStatistics()` → 获取偏离统计

### 6.3 业务逻辑层 (services/)

#### subjectService.js - 受试者管理
- `createSubject(data)` → 创建受试者并自动生成访视计划
- `getSubject(id)` → 获取受试者详情（含访视计划）
- `listSubjects(filters)` → 获取受试者列表
- `updateSubject(id, data)` → 更新受试者信息
- `deleteSubject(id)` → 删除受试者（级联删除访视记录）
- `withdrawSubject(id)` → 受试者脱落处理（停止后续提醒）
- `importFromExcel(filePath)` → Excel批量导入

#### visitService.js - 访视计划管理（核心）
- `generateVisitPlan(enrollmentDate)` → 根据入组日期生成完整访视计划
- `getVisitPlan(subjectId)` → 获取受试者访视计划
- `fillActualDate(visitId, actualDate)` → 回填实际访视日期
  - 核心逻辑：回填后自动重算后续所有访视的计划日期和窗口期
  - 检查是否超窗，若超窗自动生成偏离记录
  - 若是最后一个访视，自动将受试者状态更新为"完成"
- `recalculateFromVisit(subjectId, fromVisitOrder, newBaseDate)` → 从指定访视开始重算
- `checkAndMarkDeviations()` → 批量检查并标记方案偏离

#### reminderService.js - 提醒清单
- `getTodayReminders()` → 今日待访视列表
- `getTomorrowReminders()` → 明日待访视列表
- `getWeekReminders()` → 本周待访视列表
- `getUpcomingReminders(days)` → 未来N天待访视列表
- 所有提醒方法需：
  - 排除已脱落的受试者
  - 按窗口截止日期排序（紧急的在前）
  - 标记预警状态（剩余天数≤2天）

#### deviationService.js - 方案偏离管理
- `autoDetectDeviations()` → 自动检测方案偏离
- `getDeviationList(filters)` → 获取偏离列表
- `getDeviationSummary()` → 获取偏离汇总报表
- `generateDeviationReport()` → 生成偏离报表数据

#### excelService.js - Excel导入导出
- `parseSubjectExcel(filePath)` → 解析受试者Excel
- `exportVisitPlan(subjectIds)` → 导出访视计划
- `exportDeviationReport()` → 导出偏离报表

## 七、核心业务流程

### 7.1 访视计划生成流程
```
入组日期 → 遍历VISIT_PLAN_TEMPLATE
  → 筛选期: plannedDate = enrollmentDate, window = [D0, D0]
  → C1: plannedDate = enrollmentDate + 21, window = [D0+18, D0+24]
  → C2: plannedDate = C1_planned + 21, window = [C1_planned+18, C1_planned+24]
  → ...依次类推
→ 批量写入visits表
```

### 7.2 实际日期回填流程
```
回填actualDate → 检查是否在窗口内
  → 超窗: 标记visit.status = 'deviation', 生成偏离记录
  → 在窗: 标记visit.status = 'completed'
→ 以actualDate为新基准，重算后续所有pending访视的plannedDate和window
→ 更新数据库
→ 若为最后一个访视且已完成, 更新subject.status = 'completed'
```

### 7.3 每日提醒流程
```
获取今日日期 → 查询所有status=pending的访视
  → 过滤已脱落受试者
  → 判断今日是否在各访视窗口范围内
  → 计算剩余天数(daysUntilDeadline)
  → 按剩余天数升序排序
  → 剩余天数≤2天标记为urgent
```

## 八、API接口设计

### 8.1 受试者管理
- `POST /api/subjects` - 创建受试者
  - Body: `{ subject_no, name_abbr, enrollment_date }`
  - Response: 受试者详情 + 访视计划

- `GET /api/subjects` - 受试者列表
  - Query: `?status=active&page=1&limit=20`
  - Response: `{ list, total, page, limit }`

- `GET /api/subjects/:id` - 受试者详情
  - Response: 受试者信息 + 完整访视计划

- `PUT /api/subjects/:id` - 更新受试者
  - Body: `{ name_abbr, enrollment_date, status }`

- `DELETE /api/subjects/:id` - 删除受试者

- `POST /api/subjects/:id/withdraw` - 受试者脱落
  - 停止后续所有提醒

- `POST /api/subjects/import` - Excel导入
  - FormData: file字段
  - Response: 导入结果统计

### 8.2 访视管理
- `GET /api/subjects/:id/visits` - 获取访视计划
- `PUT /api/visits/:id/actual-date` - 回填实际日期
  - Body: `{ actual_date }`

### 8.3 提醒清单
- `GET /api/reminders/today` - 今日提醒
- `GET /api/reminders/tomorrow` - 明日提醒
- `GET /api/reminders/week` - 本周提醒
- 所有返回格式:
```json
{
  "date": "2026-06-13",
  "reminders": [
    {
      "subject_no": "S001",
      "name_abbr": "ZS",
      "visit_type": "c1",
      "visit_name": "治疗期C1",
      "planned_date": "2026-07-04",
      "window_start": "2026-07-01",
      "window_end": "2026-07-07",
      "days_remaining": 2,
      "urgency": "urgent"
    }
  ]
}
```

### 8.4 方案偏离
- `GET /api/deviations` - 偏离列表
- `GET /api/deviations/summary` - 汇总统计
- `POST /api/deviations/detect` - 手动触发检测

## 九、错误处理

### 统一错误响应格式
```json
{
  "error": {
    "code": "SUBJECT_NOT_FOUND",
    "message": "受试者不存在"
  }
}
```

### 错误码定义
| 错误码 | HTTP状态 | 说明 |
|--------|---------|------|
| VALIDATION_ERROR | 400 | 数据校验失败 |
| SUBJECT_NOT_FOUND | 404 | 受试者不存在 |
| VISIT_NOT_FOUND | 404 | 访视记录不存在 |
| DUPLICATE_SUBJECT_NO | 409 | 受试者编号重复 |
| INVALID_DATE_FORMAT | 400 | 日期格式错误 |
| IMPORT_FAILED | 500 | Excel导入失败 |
| INTERNAL_ERROR | 500 | 内部服务器错误 |
